# 领锁官网 审核优化闭环流程 PRD

**版本**: v1.0
**创建日期**: 2026-03-22
**状态**: 启用
**触发方式**: 手动 + 每周定时自动 + **10分钟极速迭代**

---

## 1. 背景与目标

### 1.1 问题陈述
领锁官网长期缺乏系统化质量审核，存在：
- 弹窗骚扰、虚假数据等用户体验问题长期存在
- CSS技术债积累（重复声明、死代码）
- 图片未优化影响加载性能
- SEO问题无人跟进

### 1.2 目标
建立**自动化闭环**：定时审核 → 问题分级 → 并行修复 → 合规复审 → 自动上线，让官网质量持续迭代无需人工干预。

### 1.3 成功指标
- 每周至少完成1次完整闭环
- 高优先级问题从发现到上线不超过24小时
- 0个高优先级问题遗留超过1周

---

## 2. 流程架构

### 2.1 流程全景

```
[触发] ─── 手动(Hugo喊"启动官网审核")
      └── 定时(每周日03:00自动)

         ↓

[阶段1] 三路并行审核 (3分钟)
  ├── 领锁码农 → 代码质量/CVE/性能
  ├── 领锁合规 → UX/弹窗/合规风险
  └── 领锁优化 → 图片/CDN/SEO
  全站扫描，给出优化方案

         ↓

[阶段2] 问题汇总分级 (并入阶段1)
  └── 生成问题清单JSON → 高/中/低分级

         ↓

[阶段3] 三路并行修复 (5分钟迭代周期)
  ├── 领锁码农 → CSS/JS技术债
  ├── 领锁合规 → 弹窗/假数据
  └── 领锁优化 → 图片/CDN

         ↓

[阶段4] 合规验证审查 (2分钟)
  └── 领锁合规验证修复效果

         ↓

[阶段5] 本地发布 → GitHub提交
         ↓
[阶段6] 飞书通知Hugo（修复内容+预览链接）
```

### 2.2 角色分工

| 角色 | 职责 | 超时 |
|------|------|------|
| 领锁码农 | CSS/JS/HTML技术债修复 | 10分钟 |
| 领锁合规 | UX/弹窗/假数据/复审 | 10分钟 |
| 领锁优化 | 图片/CDN/SEO优化 | 10分钟 |
| 领锁参谋 | 问题汇总分级协调 | 5分钟 |

## 3. 触发机制

### 3.1 手动触发
- Hugo对机器人说"启动官网审核"或"跑一轮官网闭环"
- 立即触发全流程

### 3.2 定时触发（每周审核）
- 每周日03:00自动运行
- 避免在工作时段打扰

### 3.3 极速迭代（10分钟循环）
- 每10分钟自动跑一轮
- 三路并行扫描+修复+GitHub提交
- 持续优化，形成迭代经验
- 每次必须有实质改进，空跑则跳过

---

## 4. 每轮闭环详细说明

### 阶段1 - 并行审核（15-20分钟）

**领锁码农审核清单**：
- [ ] CSS重复声明/冲突
- [ ] HTML内联样式vs CSS冲突
- [ ] 死代码（未使用的CSS类）
- [ ] 响应式边界问题
- [ ] JS性能（事件监听重复等）
- [ ] !important滥用

**领锁合规审核清单**：
- [ ] 弹窗数量>2个/页面
- [ ] 自动触发的弹窗（滚动触发/定时触发）
- [ ] 虚假社会证明数据（Math.random等）
- [ ] Cookie横幅位置
- [ ] 表单验证体验（alert()使用）
- [ ] 移动端菜单可用性

**领锁优化审核清单**：
- [ ] 图片>150KB未压缩
- [ ] 图片未使用CDN
- [ ] 首屏图片无预加载
- [ ] SEO meta标签完整性
- [ ] 页面加载性能

### 阶段2 - 问题汇总分级

**分级标准**：
- 🔴 高：直接影响用户体验/信任度/性能
- 🟡 中：影响开发效率/有劣化风险
- 🟢 低：最佳实践建议

**输出格式**：
```json
{
  "date": "2026-03-22",
  "high_priority": [
    {"id": "HP-001", "问题": "...", "修复者": "领锁码农", "文件": "index.html"}
  ],
  "medium_priority": [...],
  "low_priority": [...]
}
```

### 阶段3 - 并行修复

三路Agent同时修复，修复完成后各自汇报改动内容。

### 阶段4 - 合规复审

领锁合规对修复内容进行验证：
- 确认高优先级问题已解决
- 确认没有引入新问题
- 标记：通过 / 打回重修

### 阶段5 - GitHub提交 + Vercel部署

```bash
git add -A
git commit -m "fix: 官网审核优化 $(date +%Y-%m-%d)"
git push
# Vercel自动检测到push，自动部署
```

### 阶段6 - 飞书通知

通知内容：
- 修复了哪些问题（高/中/低分列）
- 预览链接
- 是否有遗留问题

## 5. 自动化实现

### 5.1 Cron定时任务
- Job ID: `lock-website-weekly-audit`
- 时间: `0 3 * * 0` (每周日03:00)
- 执行内容: 启动三路并行审核 → 修复 → 通知

### 5.2 核心脚本
- `/lock-club-website/scripts/audit-loop.sh` - 闭环主脚本
- `/lock-club-website/logs/` - 审核日志目录
- 问题清单: `/lock-club-website/logs/audit-report-YYYYMMDD.json`

### 5.3 审核报告存储
- 每次审核生成JSON报告
- 保留最近30天的报告
- 历史报告用于趋势分析

---

## 6. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 修复引入新问题 | 高 | 领锁合规复审兜底 |
| 多人同时修改冲突 | 中 | 修复前先git pull |
| Vercel部署失败 | 中 | 飞书通知Hugo + 保留git历史 |
| Agent修复失败 | 低 | 标记问题打回，人工介入 |
